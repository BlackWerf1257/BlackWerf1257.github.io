## 最適化

- 対象プラットフォームおよび詳細機能に関する事前調査が必須である
- プロトタイプ段階では、最適化を気にせずに開発を行う

### プロファイリング

#### 意味

- プログラムの実行速度と安定性を向上させるため、CPUやメモリの使用状況、関数の実行時間などの要素を追跡すること

<br/>

#### 注意事項

- 開発中に頻繁に行うこと

  - 後の段階で実施すると、作業の負担が大きくなる

- 推測に基づく最適化を行わない

- デバッガーなどのツールを積極的に活用する

  - 例）Unityエディタのプロファイラ

### GPUプロファイリング

- 下部に記載されているドキュメントの最新版（2022.1エディター）では、DirectX 11および12でのみ使用可能です
- 「Windows」＞「Analysis」＞「Profiler」メニューからプロファイリングウィンドウを開くことができます

#### 意味

- ゲームのどの部分がGPUをどのくらいの時間使用したかを表示するプロファイラ

- 単位：ms

  <figure>
  <img data-action="zoom" src='{{ "assets/img/Unity/GPU_Profiling_Img.png" | relative_url }}' alt='absolute'>
    <figcaption> 上の画像は、UnityにおけるGPU Profilerの様子である</figcaption>
  </figure>
  

上の写真をよく見ると、どこでGPUにDrawCallを実行したか、その回数や実行時間などが表示されています。

#### 関連する概念

- FillRate = 画面のピクセル数 × フラグメントシェーダーの複雑度 × オーバードロー

  - 画面のピクセル数：画面解像度
  - フラグメントシェーダーの複雑度：シェーダーの負荷の大きさを判断する指標
  - オーバードロー：ピクセルがどれだけ重なり合って描画されるかを示す指標

  
  
  #### メリット
 
 - プレイヤーをビルドする必要がないため、迅速なプロファイリングに適している
 
 #### デメリット
 
 - オーバーヘッドの影響を受け、結果の精度が低下する可能性がある

### テクスチャの最適化

- Unityで画像を使用する際、元のフォーマットが何であれ、GPUが処理可能なフォーマットに変換されるため、テクスチャの設定さえ適切に行えば問題はない

#### フォーマット

##### PVRTC

- サイズ：4bpp、2bpp（ピクセルあたりのビット数を選択）

- 主にiPhoneで使用される

- 原理：<br/>

  ​  1) 元の画像を4×4ブロック単位のサイズに変換して縮小する <br/>

  ​  2) 2つの代表色を選択 <br/>

  ​  3) 1:1の解像度でModulation Dataに転送し、組み合わせて処理する <br/>

- 欠点: <br/>
  - 4x4ピクセルで処理されるため、ボケが生じて画像がぼやけてしまうため、レトロスタイルのゲームでは推奨しません

##### その他

- サイズ：

  - ETC1: 4ビット/ピクセル

  - ETC2：4、8ページ

- 原理：

  - PVRTCと同様にブロック単位で処理されるが、画像を**色と輝度**に分けて合成し、処理を行う
  - 色は低解像度で、輝度は1:1の解像度で抽出・合成する

- 欠点：

  - ブロック単位で色が不自然に混ざって見える場合がある
    - ETC2ではETC1に比べてこの現象が緩和されている

  <br/><br/>

  この他にも様々な拡張子があるが、公式ドキュメントに詳しく説明されているため、リンクで代用する

  <a href="https://docs.unity3d.com/kr/2021.2/Manual/class-TextureImporterOverride.html" target="Platform Recommended Format" rel="noopener noreferrer">プラットフォーム別推奨フォーマット</a>

  <br/>

  #### 解像度
 
 - 4x4サイズで処理されるため、2^nのサイズで処理される必要がある（POT）
  - サイズの種類：4、8、16、32、64、128、256、512、1024、2048...
    - これ以外の値を入力すると、Unityが自動的にサイズを拡大して処理するため、無駄なスペースが発生する
  - ASTCでは、5x5、12x12のようにブロックのサイズを自由に選択できる

### 最適化に関するヒント

#### メモリ管理

- モバイルゲームであればあるほど、メモリ最適化の重要性が高まる

  - ETC[^1]2による圧縮を利用する際、端末がES 3.0以上に対応しているか確認する

    - OpenGL ES 2.0でETC2を使用すると、圧縮が解除され、メモリ使用量が再び増加する（4MB → 16MB）

    - ES2.0デバイスをターゲットとする場合は、ビルド設定で**ETC1を使用して圧縮**オプションを有効にしてください

      （Sprite&UIでのみ対応しており、Packing Tagが必須です）

    <br/>

  - オーディオは「Load Type」を指定することでメモリを節約できる
    - Decompress On Load：反応は速いが、読み込み後に圧縮を解いて再生するため、元のサイズそのままがメモリに読み込まれるという保証はない
 
 ​  圧縮を解く際、圧縮を維持する場合に比べて10倍以上のメモリを使用するため、容量の小さいファイルにのみ適している
    
    - Compress In Memory: メモリ上で圧縮状態を維持し、再生時に解凍する
 
 ​  解凍時にメモリを大量に消費するため、大容量のファイルにのみ使用すべきである
 
 - Streaming: サウンドを即座にデコードする方法
 
 ​  ディスクから順次読み込み、圧縮データをバッファに格納することでメモリ使用量を最小限に抑える
    
    <br/>
 
 - Unity上では同一ファイルかどうかを確認する機能がないため、重複ファイルがないよう注意すること（フォント、テクスチャなど）
    - Memory Profilerで重複ファイルの有無を確認可能
 
 - Meshを使用する際、**Read/Write Enable**オプションがオフになっているか確認すること
    - スクリプトを通じてMeshへのアクセス可否を選択するオプションであり、GPUやCPUのメモリに重複して読み込まれるため、不要な場合はオフにする
 
 [^1]: Androidの標準テクスチャフォーマット

#### アセットの管理方法

- アセットタイプ／プラットフォームごとに設定する
- 適切なテクスチャ形式を選択する
- メッシュのインポートオプションを選択する
- オーディオを圧縮して使用する



#### プログラミングとコードアーキテクチャ

- Awake、OnEnable、Updateなどのイベント関数の呼び出し順序を把握しておく
  - <a href="https://docs.unity3d.com/kr/2022.3/Manual/ExecutionOrder.html" target="イベント関数の呼び出し順序" rel="noopener noreferrer">イベント関数の呼び出し順序</a>

- 適切なTargetFrameRate
  - ジャンルや状況に応じて設定する
  - Adaptive Performanceを使用する
- 非同期読み込みを使用する
  - アセットの読み込みなどは非同期で処理することで、パフォーマンスを向上させることができる

- 使用していないMonobehaviorsメソッドを削除する
  - Start、Update、FixedUpdate……
- Updateを使用するよりも、マネージャーパターンを利用する

<br/>

- コストの高いAPI呼び出しは最小限に抑える
  - GameObject.Find()
    - transform.find、インスペクターでの指定、FindWithTag...などで代替可能
  - GameObject.GetComponent
    - TryGetComponent()の方が呼び出しコストが低い
- 文字列操作に注意する
- Debug.Log() に注意
  - Debug クラスを使用するか、Conditional で条件を指定するか、使用後に削除することで、ビルド時に除外される
- allocationg API に注意
  - s が付く API は配列として割り当てられるため、呼び出しコストが高い
- 静的データのパースに注意
  - Json/XML: スクリプタブルオブジェクト
